# 08-graceful-exit: 종료 + 최종 보고 스킬

> **실행자**: Leader (직접 실행)
> **실행 시점**: QC 통과 후 (정상) 또는 파이프라인 어느 단계에서든 실패 시
> **쓰기 권한**: `output/`, PIPELINE-LOG.md, FINAL-REPORT.md

## 목적

작업 결과(성공, 부분 성공, 전체 실패, 예산 도달 등)에 상관없이 실행 중인 에이전트와 프로세스를 안전하게 정리(Clean-up)하고 최종 요약 보고서를 작성하여 유저에게 반환합니다.
가장 중요한 원칙은 **"어떤 상황에서도 유저의 파일과 중간 산출물을 무단 삭제하지 않으며, 빈손으로 돌려보내지 않는다"**는 것입니다.

## 실행 시점

- 07-qc-validate (QC) 과정 통과 후 정상 종료 시점에 자동 실행됩니다.
- 선행되는 T1~T5 스킬 중 복구 불가능한 에러(Fatal Error)나 예산 초과(Quota Error)가 발생하여 강제 종료해야 할 때 활용합니다.

## 워크플로우

### Step 1: 상태 진단 및 시나리오 판별

- **✅ 시나리오 1 (정상 완료)**: QC 통과. 팀원 전체 수동 종료 요청 및 `output/FINAL-REPORT.md` 작성.
- **⚠️ 시나리오 2 (부분 실패)**: 특정 단계만 실패한 경우 대체 산출물 제공(예: T3 실패 시 정지-음악 슬라이드쇼, T4 실패 시 무음 클립).
- **🔴 시나리오 3 (전체 실패)**: T1 등 코어 기반부터 실패한 경우 시놉시스나 프롬프트 같은 원천 텍스트 정보만이라도 `output/`에 전달 및 실패 사유 명기.
- **🟡 시나리오 4 (예산 한도)**: `config/budget-limits.json` 위반 시 즉시 MCP 구동을 중단하고 추가 예산/진행 여부를 유저와 합의.

### Step 2: 팀 해산 (Clean-up) 및 로그 정리

- 현재 활성화된 에이전트 팀원(Teammate 1~3) 전원에게 작업 일시 중지 및 종료 지시.
- `PIPELINE-LOG.md`에 최종 도출된 상태(✅, ⚠️, 🔴, 🟡)를 최하단에 기록.

### Step 3: 최종 보고서(FINAL-REPORT.md) 생성

- `templates/FINAL-REPORT.template.md` 포맷에 따라 작성.
- 프로젝트 ID(timestamp), 도출된 결과 종류, T1~T5 소요시간 합계, 소모 비용, 보안 조치 내역(QC), 산출물의 무결성 해시(SHA-256) 삽입.
- 복구 시나리오(에러 발생 시): "project-{timestamp} T{N}부터 재실행" 등의 리커버리 커맨드 가이드라인 명시.

### Step 4: 에셋 아카이빙(보존)

- `input/` : 제공받은 원본 전체.
- `work/` : 실패 여부 무관 T1~T5 에셋 중간본 유지.
- `quarantine/` : 안전성(QC) 탈락본 격리 샌드박스 유지.

## 관련 파일

| File | Purpose |
|------|---------|
| `07-qc-validate.md` | 정상적인 파이프라인의 직전 단계 검수 스킬 |
| `PIPELINE-LOG.md` | 종합 로그 파이프라인 기록 마커 갱신 위치 |
| `templates/FINAL-REPORT.template.md` | 생성될 최종 보고서의 스키마 구조 규격 |

## 예외사항

1. 무응답/고스트 에러: 팀원이 응답오류에 빠졌더라도, 강제 파일 IO(강제 다운그레이드 아카이브)로 현재까지 디렉토리에 생성된 파일을 긁어와서라도 리포트를 만듭니다.
2. 예산 도달 이후 유저 강제 속행: 유저가 로그오류 또는 비용 초과를 감수하고 "일단 계속 생성해"라고 지시할 경우 부분 산출물을 넘기고 재시작 모드로 이관합니다.
