# ToT / ToC / Reflection 적용 가이드

## 기법 요약

| 기법 | 설명 | 적용 지점 |
|------|------|----------|
| **Tree-of-Thought (ToT)** | 하나의 문제에 대해 여러 사고 경로를 동시 전개 | Orchestrator 태스크 분해, Skill Forge 구조 설계 |
| **Chain-of-Thought (CoT)** | 단계별 순차 추론 | Instruction Gen의 system prompt 내 scaffold |
| **ToC Scoring** | 분기별 가중 평가 점수로 최적 경로 선택 | Orchestrator 분기 선택 시 |
| **Reflection** | 자체 출력물에 대한 비판적 리뷰 | Verification Layer Stage 3 |
| **Iterative Refinement** | 검증 실패 시 수정 후 재검증 | Verification 자동 수정 루프 |

---

## ToT — 단일 프롬프트 패턴 (API 호출 0)

### 왜 단일 프롬프트인가

원래 ToT는 각 분기를 별도 API 호출로 생성합니다.
하지만 우리 환경은 **MAX_TOKENS 10,000 + 외부 API 금지**이므로,
단일 추론 과정에서 3분기를 동시 생성하는 **인라인 ToT**를 사용합니다.

### 인라인 ToT 프레임워크

```markdown
## 분석 대상: {요청}

### 사고 분기 A — [전략명]
핵심 아이디어: ...
장점: ...
단점: ...
예상 비용: 토큰 ~X

### 사고 분기 B — [전략명]
핵심 아이디어: ...
장점: ...
단점: ...
예상 비용: 토큰 ~Y

### 사고 분기 C — [전략명]
핵심 아이디어: ...
장점: ...
단점: ...
예상 비용: 토큰 ~Z

### 평가
| 분기 | 실현 | 품질 | 비용 | 총점 |
|------|------|------|------|------|
|  A   |  ?   |  ?   |  ?   |  ?   |

→ 선택: Branch {X}, 사유: {1줄}
```

## ToC Scoring — 가중 점수 계산

### 가중치 공식

```
총점 = 실현가능성 × 0.4 + 품질 × 0.35 + 토큰비용(역산) × 0.25
```

### 토큰비용 역산 규칙

```
10점: ~1,000 토큰 이하
 8점: ~2,000 토큰
 6점: ~3,000 토큰
 4점: ~5,000 토큰
 2점: ~8,000 토큰
 1점: MAX_TOKENS 초과 위험
```

## Reflection — 자체 검토 패턴

### Self-Review 프롬프트 (Verification Layer에서 사용)

```markdown
다음 출력물을 검토하라:
[출력물]

검토 기준:
1. 원본 요청과의 일치도 (0-10)
2. 에지케이스 처리 여부
3. 토큰 효율성
4. 보안/안전성

결과:
| 항목 | 심각도 | 설명 | 수정안 |
|------|--------|------|--------|
```

### Reflection 토큰 절약 전략

- **simple** 태스크: Reflection 스킵
- **composite** 태스크: 생성 후 인라인 검토 (같은 출력에서)
- **research** 태스크: 별도 Reflection 단계 (verification-layer 호출)

## 예시: Slack 리포트 스킬 ToT 전개

```
요청: "Slack 메시지를 분석해서 주간 리포트 만드는 스킬 만들어줘"

Branch A: 최소 접근
  → Slack MCP만 사용, 단일 SKILL.md에 분석+리포트 통합
  → 장점: 토큰 절약, 유지보수 쉬움
  → 단점: 기능이 단순할 수 있음

Branch B: 조합 접근
  → Slack MCP + 별도 분석 스크립트 + 문서 생성 스킬
  → 장점: 각 모듈이 독립적, 재사용 가능
  → 단점: 토큰 소모 많음, 모듈 간 연동 복잡

Branch C: 기존 리소스 재활용
  → 기존 prompts 파이프라인의 에이전트를 fork
  → 장점: 검증된 구조 활용
  → 단점: 원래 용도(크랙 프롬프트)와 맞지 않을 수 있음

평가:
  A: 8×0.4 + 7×0.35 + 9×0.25 = 7.90
  B: 6×0.4 + 8×0.35 + 5×0.25 = 6.45
  C: 5×0.4 + 6×0.35 + 8×0.25 = 6.10

→ Branch A 선택
```
